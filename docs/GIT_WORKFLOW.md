## 1. Overview: The Two-Commit Strategy

| Commit | When | What it captures |
| :----- | :--- | :--------------- |
| **Commit 1** | After cloning and copying source code (before any changes) | Raw source code exactly as it came from the forked repo |
| **Commit 2** | After all fixes are applied and the app is verified working | Every change made to get the application running |

Anyone who clones your repo can run `git diff HEAD~1 HEAD` to see the exact diff — a complete record of what had to be fixed.

## 2. Commit 1: Initial Source Code Push

> **When to do this:** Immediately after completing **Phase 2: Clone & Organise** in `docs/DEVELOPER_SETUP.md`, before installing system software, configuring `.env`, or making any code changes.

---

### Step 1: Configure Git

Verify Git is accessible:

```bash
git --version
```

**Expected output:**

```
git version 2.54.0
```

Configure your identity — required for all commits:

```bash
git config --global user.name "Your Name"

git config --global user.email "your-email@example.com"
```

Set the default branch name to `main`:

```bash
git config --global init.defaultBranch main
```

Verify your configuration:

```bash
git config --list
```

**Expected output includes:**

```
user.name=Your Name
user.email=your-email@example.com
init.defaultbranch=main
```

**Verification:** ✅ Git configured

---

### Step 2: Initialise Git Repository

```bash
cd ~/humor-memory-game

git init
```

**Expected output:**

```
Initialized empty Git repository in /home/developer/humor-memory-game/.git/
```

**Verification:** ✅ Git repository initialised

---

### Step 3: Create `.gitignore`

This prevents secrets, generated artefacts, and OS files from ever being committed.

```bash
cat > ~/humor-memory-game/.gitignore << 'EOF'
# Dependencies - generated by npm install
node_modules/

# Build artifacts - generated by npm run build
dist/
build/

# Environment secrets - NEVER commit
.env
.env.local
.env.*.local

# IDE and editor files
.vscode/
.idea/
*.swp
*.swo
*~

# Logs
*.log
npm-debug.log*
yarn-debug.log*

# OS files
.DS_Store
Thumbs.db

# Cache
.cache
tmp/
temp/
EOF

cat ~/humor-memory-game/.gitignore
```

**Verification:** ✅ `.gitignore` created

---

### Step 4: Verify `.env` is Excluded

Confirm the exclusion rule is present before staging anything:

```bash
grep "\.env" ~/humor-memory-game/.gitignore
```

**Expected output:**

```
.env
.env.local
.env.*.local
```

**Verification:** ✅ Secrets will not be committed

---

### Step 5: Stage and Commit

```bash
cd ~/humor-memory-game

git add .

git status
```

**Should be staged (✅):**

```
backend/
frontend/
database/
.gitignore
```

**Should NOT appear (❌):**

```
.env
node_modules/
dist/
```

Create the initial commit:

```bash
git commit -m "Initial commit: raw source code from forked repo — pre-configuration"
```

**Expected output:**

```
[main (root-commit) xxxxxxx] Initial commit: raw source code from forked repo — pre-configuration
 XX files changed, XXXX insertions(+)
```

Verify:

```bash
git log --oneline
```

**Expected output:**

```
xxxxxxx (HEAD -> main) Initial commit: raw source code from forked repo — pre-configuration
```

**Verification:** ✅ Initial commit created on local repository

---

### Step 6: Create GitHub Repository

Verify you are authenticated:

```bash
gh auth status
```

**Expected output:**

```
github.com
  ✓ Logged in to github.com account <USERNAME>
  - Active account: true
  - Git operations protocol: ssh
```

#### Option A — GitHub CLI (recommended)

Create the repository, add the remote, and push in one command:

```bash
gh repo create humor-memory-game \
  --public \
  --description "Full-stack memory card game - DevOps learning project" \
  --source . \
  --push
```

**Expected output:**

```
✓ Added remote git@github.com:<USERNAME>/humor-memory-game.git
...
✓ Pushed commits to git@github.com:<USERNAME>/humor-memory-game.git
```

#### Option B — GitHub Web Interface

1. Open <https://github.com/new>
2. **Repository name:** `humor-memory-game`
3. **Description:** Full-stack memory card game - DevOps learning project
4. **Visibility:** Public
5. **Important:** Do NOT check "Add README" or ".gitignore" — these already exist locally
6. Click **Create repository**

Then connect the remote manually:

```bash
git remote add origin git@github.com:YOUR-USERNAME/humor-memory-game.git
```

---

### Step 7: Push to GitHub

```bash
git push -u origin main
```

**Verification:** ✅ Initial source code pushed to GitHub

## 3. Commit 2: Working Application Push

> **When to do this:** After completing **Phase 8: Verification Testing** in `docs/DEVELOPER_SETUP.md` and confirming all tests pass.

At this point your working directory contains all the changes made during setup — the patched `combined-init.sql`, the `build-local.sh` script, the fixed `game.js` route order, the `.env` file (excluded by `.gitignore`), and the `diagnose.sh` script.

---

### Copy DOCS from host to VM

From VM create a docs directory
```bash
cd humor-memory-game

mkdir docs
```

Using SCP (Command Line)

Open your local terminal and run the following command:

```bash
scp /path/to/local/file.txt username@vm_ip_address:/path/to/remote/destination/

# Example
scp ~/"KubeCraft Example Vault/1 Projects/Devops Projects/Humor Memory Game Project/2.0 Developer Setup Guide/GIT_WORKFLOW.md" developer@192.168.30.10:~/humor-memory-game/docs/
```


### Step 1: Stage All Changes

```bash
cd ~/humor-memory-game

git add .

git status
```

**Expected staged files:**

```text
modified:   backend/package-lock.json
modified:   backend/package.json
modified:   backend/routes/game.js
modified:   database/combined-init.sql
new file:   frontend/build-local.sh
modified:   frontend/package-lock.json
```

> **Tip:** If you see `node_modules/` or `dist/` in the staged list, your `.gitignore` is not working. Stop and check it before proceeding.

---

### Step 2: Review What Changed

Before committing, confirm the diff looks correct:

```bash
git diff --cached --stat
```

This shows a summary of every file changed and lines added or removed. Review it to make sure no unexpected files are included.

For a full diff of a specific file:

```bash
git diff --cached backend/routes/game.js
```

---

### Step 3: Verify README.md is Present

Before committing, confirm `README.md` exists at the root of `~/humor-memory-game/`. This file must be present before the working state is committed.

```bash
ls ~/humor-memory-game/README.md
```

**Expected output:**

```
/home/developer/humor-memory-game/README.md
```

Stage the README along with the other changes:

```bash
git add README.md
```

---


### Step 4: Commit the Working State

```bash
git commit -m "Working application: env configured, schema patched, route collision fixed, build script added"
```

Feel free to adjust the message to describe what you specifically changed. A clear message helps anyone (including your future self) understand what this commit represents.

**Expected output:**

```
[main xxxxxxx] Working application: env configured, schema patched, route collision fixed, build script added
 X files changed, XX insertions(+), XX deletions(-)
```

---

### Step 5: Push to GitHub

```bash
git push
```

**Expected output:**

```
Enumerating objects: X, done.
...
To git@github.com:<USERNAME>/humor-memory-game.git
   xxxxxxx..xxxxxxx  main -> main
```

**Verification:** ✅ Working application pushed to GitHub

---

## 4. Verify Your Commit History

```bash
git log --oneline
```

**Expected output — two commits:**

```
xxxxxxx (HEAD -> main, origin/main) Working application: env configured, schema patched, route collision fixed, build script added
xxxxxxx Initial commit: raw source code from forked repo — pre-configuration
```

To see the full diff between the two commits — everything that changed to get the app working:

```bash
git diff HEAD~1 HEAD
```

Or a summary of changed files only:

```bash
git diff HEAD~1 HEAD --stat
```

**Verification:** ✅ Two-commit history showing before and after states
